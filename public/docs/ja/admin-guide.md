# ArchivistPro 管理者ガイド

> バージョン 3.0.0 | 2026-05-23 — このガイドは Admin (フル権限) ユーザー専用です。

> **管理者向け v3.0.0 サマリー:** このリリースでは **V3 スキーマアーキテクチャ** が導入されます — ベクターデータ (embeddings、テキストチャンク、関係、FTS) は別ファイル `archivist_vec.db` に移動されました。既存のアーカイブは初回起動時に自動移行されます (`.bak` バックアップ付き)。新しいファイル構造に合わせてバックアップワークフローを更新してください。下記「4a. V3 スキーマアーキテクチャ」を参照。

## 1. ユーザーロールと権限階層

ArchivistPro には 3 段階の権限階層があります:

| ロール | 誰か | 概要 |
|-----|--------|--------|
| 創設管理者 (Founder Admin) | プログラムを最初にインストールした、最初に作成された admin アカウント | すべてを実行可能 + 他の管理者を管理 |
| 管理者 (Admin) | 創設者によって追加された管理者 | アーカイブと viewer ユーザーを管理 |
| ユーザー (Viewer) | 通常のユーザー | アーカイブは読み取り専用、自分のローカルアーカイブを所有 |

### 権限テーブル

| 操作 | 創設者 | 管理者 | ユーザー |
|-------|:------:|:--------:|:---------:|
| アーカイブ表示 | はい | はい | はい |
| アーカイブ書き込み / スキャン | はい | はい | いいえ |
| Viewer 追加 / 削除 | はい | はい | いいえ |
| 新規管理者 (admin) 追加 | はい | いいえ | いいえ |
| 管理者削除 / ロール降格 | はい | いいえ | いいえ |
| アプリケーション設定の管理 | はい | はい | いいえ |
| ログ表示 | はい | はい | いいえ |
| 創設者のロール / 削除 | いいえ | いいえ | いいえ |

> **創設者とは誰か?** プログラムの初回起動時に「初期セットアップ」画面で作成された admin アカウントです。
> 自動的に割り当てられ、このアカウントは削除もロール降格もできません。
> 別のコンピューターに移行する場合でも、保護されるアカウントは同じ DB 内の最初の admin として残ります。

### サンプルシナリオ

2 人の管理者 (Ayşe = 創設者、Mehmet = 後で追加) が同じオフィスで働いています:
- Mehmet は viewer ユーザーを追加・削除できます (フル権限)
- Mehmet は Ayşe を削除したり、ロールを降格したりできません
- Mehmet は新しい admin を追加できません — この決定は Ayşe の権限です
- Ayşe は希望すれば Mehmet のロールを viewer に降格できます

## 2. 初回起動ウィザード (Setup Wizard) と Admin アカウント

アプリケーションが初めて開かれると、2 段階のセットアッププロセスが発生します:

### 2a. セットアップウィザード

ログイン画面の前に、**セットアップウィザード (Setup Wizard)** が自動的に表示されます。このウィザードは 4 つのステップでユーザーをシステムに準備します:

1. **ウェルカム & システムチェック**
   - WebAssembly (WASM) サポートが確認されます
   - OS バージョンと推定ディスク容量が表示されます
   - **言語選択** が可能です (Türkçe / English) — ログイン画面の前なので、ここで提示されます

2. **ハードウェア検出**
   - CPU コア数、RAM、ベンチマーク結果が表示されます
   - システムは自動的に Low / Medium / High パフォーマンスモードを推奨します
   - 必要に応じて別のモードを選択できます

3. **AI セットアップ**
   - Ollama サーバーが自動的にチェックされます (localhost:11434)
   - Ollama が稼働していれば、利用可能な vision モデルがリストされます
   - 3 つの選択肢が提示されます:
     - **ローカル AI (Ollama)** — データはローカルに残ります、GPU が推奨されます
     - **クラウド AI (Gemini/Groq)** — インターネットと API キーが必要です
     - **AI をスキップ** — 後で設定から有効化できます
   - 「再チェック」ボタンで Ollama のステータスを再クエリできます

4. **準備完了!**
   - 選択したパフォーマンスモードと AI モードがまとめられます
   - 「Archivist Pro を起動」ボタンで次のステップに進みます

### 2b. 最初の Admin アカウント作成

ウィザードが完了した後、**まだユーザーが作成されていない場合**、ログイン画面の代わりに **「初期セットアップ (First Setup)」** 画面が開きます:

- ユーザー名とパスワードを設定します (ユーザー名は 3〜32 文字、パスワードは最大 128 文字)
- 「Admin アカウントを作成」ボタンをクリックします
- アカウントが作成されると自動的にログインされます

> **重要:** システムには `admin / admin` のような既製のアカウントはありません。最初の admin アカウントはあなたが作成します。

### 既存ユーザー

以前にアプリケーションを使用したことがある人 (ウィザードが完了し、ユーザーテーブルに少なくとも 1 つのレコードがある) は、ウィザードと初期セットアップ画面を **再度見ることはありません**。

### ウィザードのリセット

テスト目的でウィザードを再表示したい場合は、ブラウザの開発者ツール (F12 > Application > Local Storage) から `archivist_setup_wizard_done` および `archivist_perf_setup_done` キーを削除します。

## 3. パスワード復旧

admin パスワードを忘れた場合、アプリケーション起動時に自動生成される `recovery.key` ファイルを使用してパスワードをリセットできます。

### Recovery Key ファイル

- 場所: `%APPDATA%\com.archivistpro.desktop\recovery.key`
- 48 文字の hex 文字列、アプリケーション初回起動時に一度だけ生成されます
- このファイルを安全な場所にバックアップしてください (USB、暗号化されたクラウドなど)

### パスワードリセット手順

1. ログイン画面の **「パスワードを忘れた」** リンクをクリックします
2. `recovery.key` ファイルの内容を入力ボックスに貼り付けます
3. パスワードをリセットする admin アカウントを選択します
4. 新しいパスワードを設定して確認します
5. ログイン画面にリダイレクトされます

> **メモ:** Recovery key は使い捨てではありません。各リセットで同じキーが有効です。キーファイルが失われ、すべてのパスワードを忘れた場合は、データベースファイルをバックアップしてアプリケーションをリセットする必要があります。

## 4a. V3 スキーマアーキテクチャ (3.0.0+)

![V3 移行パネル](img/settings-v3-migration.png)

v3.0.0 以降、ArchivistPro はアーカイブデータを**2 つの別個の SQLite
ファイル**に保存します:

| ファイル | 内容 | 一般的なサイズ |
|---|---|---|
| `archivist.db` | アセットメタデータ、タグ、コレクション、設定、ユーザー、関係タイプ | 50-100 MB |
| `archivist_vec.db` | embedding ベクター、テキストチャンク、アセット関係、FTS5 キーワードインデックス | 100-300 MB |

### なぜ分離するのか？

- **パフォーマンス:** メイン `archivist.db` が 3-4 倍小さくなる → sql.js
  のロードと書き込みが高速化。実測: 181 MB → 52 MB。
- **メモリ:** sql.js は DB をすべて RAM にロードします。小さいメイン DB =
  メモリ使用量が少ない、ブロッキング保存が少ない。
- **スケール:** 数十万の embeddings と数千のテキストチャンクが HNSW ANN
  インデックス付きの専用構造に存在。メタデータ DB は高速のまま。
- **保守性:** ベクターデータは再インデックス可能、メタデータは不可。
  分離によりバックアップ戦略が柔軟になります。

### 自動移行

以前のバージョンからのアーカイブは初回起動時に自動移行されます:

1. 起動時、`PRAGMA user_version` が読み取られる。
2. `user_version < 3` の場合、`runV3EpochMigration` がトリガー。
3. **プリ移行バックアップ:** `archivist.db` が `archivist_premigrate_v3.db.bak`
   にコピーされる (ロールバック用)。
4. embeddings / text_chunks / relations が `archivist_vec.db` に移動
   (エポック 1 → 2 → 3 の順)。
5. 検証: ソースとデスティネーションの行数と内容が完全一致するか？
6. 成功時: メイン DB の古いテーブルが `DROP TABLE` + `VACUUM` +
   `PRAGMA user_version = 3` で削除される。
7. ディスクへの原子書き込み。

> 移行は **Rust 側で**実行されます (rusqlite)。180+ MB のモノリスは
> Tauri IPC ペイロード制限のリスクがありません。

### 手動トリガー (管理者パネル)

自動移行が失敗するか、管理者の制御を優先する場合:

**設定 → ストレージ → V3 スキーマ移行** パネル (管理者のみ):

- 現在のエポック (0/1/2/3) とターゲット (3) を表示。
- 「移行開始」ボタン — 管理者がトリガーする手動フロー。
- ステップごとの進行: プリ移行 → 1/3 → 2/3 → 3/3 → 終了 → 再ロード。
- エラー時: 停止、メッセージ表示、データは無傷 (`.bak` 経由でロールバック可能)。

### オプトアウト

開発/テストのため V3 動作をフラグで無効化できます:

```js
localStorage.setItem('ARCHIVIST_V3_EPOCH', 'off');
```

`off` の場合、自動移行はトリガーされません。**初回移行の前に設定された場合のみ意味があります**;
成功した移行後の復元には `.bak` ファイルの復元が必要です。

### ロールバック

移行後に問題が発見された場合:

1. アプリケーションを閉じる。
2. `archivist_vec.db` を移動または削除する。
3. `archivist_premigrate_v3.db.bak` を `archivist.db` にリネーム
   (先に現在の `archivist.db` をバックアップ)。
4. アプリケーションを再オープン — 古い状態 (epoch=0) に戻る。

### WAL ジャーナルとネットワーク共有 (UNC)

v3.0.0 では WAL ジャーナルモードが**デフォルトで有効** — ローカルディスクで
書き込みパフォーマンスが約 30% 向上。ネットワーク共有は自動検出され
DELETE モードにフォールバックします:

- **UNC パス (`\\server\share\archive\...`):** WAL は安全でない (NetBIOS
  ロックセマンティクス) → DELETE モード強制。
- **マップされたネットワークドライブ (`Z:\archive\...` だが実際は UNC):**
  `GetDriveTypeW` で検出、DELETE モード強制。
- **ローカルディスク:** WAL は有効のまま。

env 変数でオーバーライド:

```cmd
set ARCHIVIST_DB_JOURNAL=delete
```

UNC で積極的に作業している場合、**共有を直接 UNC として開く** (マップされた
ドライブレターを使わない) — Windows NetBIOS ジャンクションはエッジケース。
大規模なマルチユーザーセットアップでは、このモードは HTTP LAN 共有の
代替として考えられます。

### マルチアーカイブ (Phase 1–3)
ArchivistPro は N 個のカスタムアーカイブを並列で管理できます。各アーカイブは別々の SQLite ファイルに保存され、独自のソースフォルダー、タグ、お気に入りを持ちます。

**新しいアーカイブの作成:**
- 設定 > アーカイブ > 「新規アーカイブ」ボタンをクリック
- 名前、タイプ (shared/personal)、オプションのディスクパスを入力
- 新しいアーカイブが左パネルのセレクターリストに追加されます

**結合 (Join/Merge):**
- 2 つのアーカイブを 1 つのアーカイブに結合します
- 操作前に両方のアーカイブのスナップショットが取得されます
- プレビュー: 結合されるアセット数、競合、タグ、エンベディング
- 競合戦略: skip / overwrite / rename
- ロールバック: 失敗した場合は完全なロールバック

**抽出 (Extract):**
- フィルターを適用してアーカイブからサブセットを作成します
- 基準: 種別、タグ、カテゴリー、日付範囲など
- モード: copy (ソースに残る) または move (ソースから削除)
- Move モードでは、ソーススナップショットによるロールバック保証があります

### ソースフォルダー管理 (サイドバーパネル)
左パネルの「ソースフォルダー」セクションは、各アーカイブのスキャン済みルートディレクトリを表示します:

- **追加:** スキャン時に自動的に保存されます (各スキャン = 別の行、フルパスマッチ)
- **再スキャン:** 3 点メニューから — そのフォルダーのみをスコープで再スキャンし、他のフォルダーには触れません
- **名前変更:** ビジュアルラベルを変更します (パスは変わりません)
- **削除:** フォルダーをリストから削除し、アセットはアーカイブに残ります
- **ファイルごと削除:** フォルダー + 配下のすべてのアセットレコードを削除します

> **メモ:** カウントはライブで計算されます (BAK と削除されたファイルは除外)。

### スキャンとインデックス作成
1. 左パネルの「フォルダーをスキャン & インデックス」ボタンをクリックします
2. モードを選択します:
   - **リストに追加** (デフォルト) — 新しいファイルが追加されます
   - **ゼロからスキャン** — *選択したフォルダー配下* のレコードのみが削除され、再スキャンされます。他のソースフォルダーには触れません (scoped replace)。
3. スキャン前に自動 DB スナップショットが取得されます (安全のため)
4. 直近 5 つのスナップショットが保持され、最も古いものは自動的に削除されます
5. スキャン中、**チェックポイント** システムが有効です — N ファイルごと (デフォルト 50) にデータがディスクに書き込まれ、クラッシュ/停電時でもスキャン済みのファイルは失われません
6. チェックポイント頻度は **設定 > ストレージ** タブから変更できます (1、5、10、25、50、75、または 100 ファイルごと)

### スキャンレポート
各スキャン後、スキップされたファイルやエラーが発生したファイルのリストが保存されます。これらのレポートには **ソースフォルダー > 3 点メニュー > 「スキャンレポート」** でアクセスできます。レポートはアプリケーションデータフォルダーに TXT としても保存されます。

### フォルダー変更検出 (Watch Folders)
設定で **「フォルダーの変更を監視」** オプションを有効にすると:
- スキャン済みフォルダーでファイルが追加/変更/削除されたときに通知を受け取ります
- **自動再スキャン** オプションも有効にすると、変更が検出されたときに該当フォルダーが自動的に再スキャンされます
- この機能は開いているソースフォルダーにのみ適用されます

### DWG メタデータ — ODA File Converter 統合
DWG ファイルから実際のメタデータ (レイヤー、ブロック、テキスト内容、xref、図面プロパティ) を抽出するために、**ODA File Converter** が使用されます。

**インストール:**
- 設定 > AI > 「ODA File Converter」セクションで自動検出が有効
- システムにインストールされている場合、Registry + PATH から自動的に検出されます
- インストールされていない場合、「ダウンロードしてインストール」ボタン (バンドルされたインストーラーまたは winget/web)

**動作:**
- スキャン中、各 DWG に対して ODA がバックグラウンドで **非表示** で実行されます (PowerShell `Start-Process -WindowStyle Hidden` ラッパーのおかげで、ウィンドウは開かず、フォーカスも奪いません)
- DWG → temp DXF に変換され、DXF パーサーでメタデータが抽出されます
- 結果は `dwgLayers`、`dwgBlockNames`、`dwgTextContents`、`dwgXrefNames` フィールドに書き込まれます
- **シェイプデータ** — ファイル内の幾何学的形状 (polyline、arc、長方形) が抽出され、シェイプ検索で使用されます
- **埋め込み OLE オブジェクト** — DWG 内に埋め込まれた Excel/Word/PDF ファイルが検出されます
- ODA がない場合、raw binary scan フォールバックに落ちます (サイレント、メタデータ不足)

**DWG 構造類似性検索:**
DWG ファイルを右クリックして「類似を検索」を選択すると、CLIP ビジュアル比較の代わりに 5 次元の **複合スコア (composite score)** が使用されます: レイヤー構造、ブロック構造、テキスト内容、シェイプデータ、pHash。この方法は CAD ファイルにより信頼できる結果を提供します。

### ファイル再編成 (Refile)
1. トップバーの「整理 (Organize)」ボタンをクリックします
2. 整理戦略を選択:
   - プロジェクト別
   - カテゴリー別
   - フェーズ別
   - 素材別
3. プレビューを確認します
4. 「適用」ボタンでファイルを移動します

## 6. バックアップとアーカイブ共有

### DB スナップショット

スナップショットはデータベースの完全なバックアップです。スキャン/インデックス作成のたびに **自動的に** 取得されます。手動でも取得できます。

> **v3.0.0 注:** スナップショットは現在、**`archivist.db` と `archivist_vec.db` を一緒に**保存します。復元は両方のファイルを原子的に復元します — V3 データ (embeddings/chunks/relations) がスナップショットから失われることはありません。

**自動動作:**
- 各スキャン開始前に静かに取得されます
- 直近 5 つのスナップショットが保持され、最も古いものは自動的に削除されます

**手動スナップショット (設定 > ストレージ):**
1. 設定を開く → 「ストレージ」タブに切り替える
2. **「バックアップを取得」** ボタンをクリック — スナップショットが日付とサイズとともにリストに追加されます
3. 各スナップショット行で:
   - **復元** — 現在のデータベースをこのスナップショットで置き換えます (確認が必要)
   - **削除** — このスナップショットのみを永続的に削除します

> **メモ:** Viewer ユーザーはメインアーカイブでスナップショットを取得できません。ローカルアーカイブではフル権限です。

### アーカイブ Export/Import (.archivistpro)

アーカイブ全体を 1 つの `.archivistpro` ファイルとしてエクスポートしたり、別のコンピューターからのアーカイブをインポートしたりできます。

**エクスポート (Export):**
1. 設定 > ネットワークタブに移動
2. 「エクスポート / レポート」ボタンをクリック
3. 保存場所とファイル名を選択
4. アーカイブが `.archivistpro` 形式 (ZIP) で作成されます

**インポート (Import):**
1. 設定 > ネットワークタブに移動
2. 「Import (.archivistpro)」ボタンをクリック
3. インポートするファイルを選択
4. マニフェストプレビューが表示されます (バージョン、アセット数、DB サイズ)
5. 確認 — 既存のデータベースは自動的に `.bak` でバックアップされます

### LAN 共有 (ミニ HTTP サーバー)

社内 LAN 経由で他のコンピューターとアーカイブを共有できます。インターネット接続は不要です。

**サーバー起動:**
1. 設定 > ネットワーク > 「サーバーを起動」ボタンをクリック
2. 画面に表示される情報をメモします:
   - **IP アドレス** (例: `192.168.1.106`)
   - **ポート** (`9471`)
   - **接続コード** (8 桁、例: `25930014`)
3. これらの情報を Viewer ユーザーと口頭/書面で共有します

**セキュリティ:**
- 接続コードは初回サーバー起動時にランダムに生成され (CSPRNG)、**永続的に保存されます** — アプリケーションを再起動してもコードは変わりません
- 「コードを更新」ボタンをクリックすると、コードを手動で更新できます (新しいコードは即座にアクティブになり、サーバーの再起動は不要です)
- コードなしではアクセスできません (403 エラー)
- 5 回失敗後、ソース IP は 5 分間ブロックされます
- サーバーはローカルネットワーク (LAN) でのみアクセス可能
- データは暗号化されません — 信頼できるオフィスネットワークでのみ使用してください

**サーバー停止:**
- 「サーバーを停止」ボタンをクリック
- すべての接続が切断されます

> **メモ:** サーバー稼働中はクライアントモード (接続) に切り替えられません。先にサーバーを停止してください。

## 7. 3ds Max バージョン変換

### クイックモード (スタンプ変更)
- ファイルのバージョンスタンプを変更します
- オリジナルファイルは保存され、隣に新しいファイルが作成されます
- 高速ですが、場合によっては問題が発生することがあります

### リアルモード (MAXScript)
- インストールされている 3ds Max を使用してファイルを再保存します
- より信頼できますが、Max のインストールが必要です
- Max はバックグラウンド (headless) で実行されます

### FBX / OBJ エクスポート
MAX ファイルを FBX または OBJ 形式に変換できます。詳細パネルで MAX ファイルが選択されている場合、2 つのエクスポートボタンが表示されます:

| モード | 説明 | 要件 |
|-----|----------|------------|
| **クイックモード** | 基本的なジオメトリ変換 | なし |
| **リアルモード** | 3ds Max ネイティブ FBXEXP/ObjExp プラグインで | 3ds Max インストール |

- リアルモードでは、お使いのコンピューターの 3ds Max インストールが自動的に検出されます (Registry スキャン)
- 変換されたファイルはダウンロードフォルダーに保存されます
- 操作中、Max はバックグラウンド (headless) で実行され、5 分のタイムアウトが適用されます

### MAX メタデータ表示
詳細パネルでは、MAX ファイル用に追加情報が表示されます:
- **レイヤー** — ファイルのレイヤー構造 (色付きラベル)
- **オブジェクト** — ファイル内のオブジェクト名 (最大 30 表示、それ以上は「+N more」のメモ)
- これらの情報は CFB binary stream から UTF-16LE として抽出されます

## 9. データベースセキュリティ

- 削除操作はアトミック (transaction で保護されています)
- Foreign key cascade が有効 — アセットが削除されると、エンベディング、タグ、お気に入りも自動的にクリアされます
- データベース保存エラーが発生した場合、通知が表示されます
- データベースパスを変更するときは、path traversal 保護が適用されます
- **v3.0.0:** 移行やディスク書き込みが失敗した場合、フローは停止し、メモリ内状態がディスクと再同期されます (偽成功を防止)
- **v3.0.0:** アーカイブ単位の書き込みロック — 複数のアーカイブを同時に使用するときの書き込み競合なし
- **v3.0.0:** Stale WAL クリーンアップ — blob-overwrite 後の孤立した `-wal`/`-shm` ファイルは自動的に削除されます

## 9b. 承認キュー (Approval Workflow)

Dashboard ビューには、管理者向けの **承認キュー** パネルがあります。このパネルでは、「レビュー中」状態のファイルを一括管理できます。

### Dashboard パネル
- **4 つのステータスバッジ:** ドラフト、レビュー中、承認済み、却下 — それぞれ何ファイルあるかが表示されます
- **保留リスト:** 「レビュー中」のファイルが並びます (最大 20、スクロール可能)
- **一括操作:** 「すべて承認」または「すべて却下」ボタン

### 却下理由
ファイルを却下すると、開かれるテキストフィールドに理由を入力できます (例: 「スケールが間違っています」)。この情報はファイルの詳細パネルおよび XMP サイドカーエクスポートに表示されます。後でファイルが承認されると、却下理由は自動的にクリアされます。

### 承認履歴 (Audit Trail)
すべての承認ステータス変更は `approval_log` テーブルに記録されます:
- 誰が変更したか
- いつ
- どのステータスから → どのステータスへ
- 却下理由がある場合は理由

Dashboard の **「承認履歴」** パネルには、最新 10 件の操作が時系列で表示されます。

## 9f. AI 感度フィルター

アーカイブ内の機密ファイル (契約書、給与表、個人データ) が AI チャットで結果として表示されないようにする 3 層保護システム。

### なぜ必要か?

AI チャットはスキャンされたすべてのファイルをクエリできます。viewer ユーザーが「クライアント契約書はありますか?」または「給与表を検索」と尋ねたとき、アーカイブにそのようなファイルがあれば AI はそれを見つけて表示します。このフィルターを使うと、機密データを AI から非表示にできます — ファイルはアーカイブには引き続き表示されますが、AI だけがアクセスできなくなります。

### レイヤー 1 — 既製のカテゴリー

設定 > セキュリティ > **AI 感度フィルター** カードから有効化します。4 つのカテゴリーをオン/オフできます:

| カテゴリー | 検出されるキーワード |
|----------|---------------------|
| **財務** | 給与、請求書、見積、予算、支払、コスト、進捗払い、見積、収入、支出、銀行、IBAN |
| **個人情報** | 身分証明書、戸籍、電話、住所、生年月日、運転免許、パスポート |
| **法務** | 契約、NDA、機密、裁判所、催告書、委任状、公証、訴訟 |
| **人事** | 人事ファイル、休暇、記録、業績、懲戒、採用、面接 |

これらのキーワードは、ファイル名、プロジェクト名、ファイル内容 (chunk text) でスキャンされます。一致するファイルは AI から完全に除外されます。

### レイヤー 2 — カスタムキーワード

同じ設定カードから独自のキーワードを追加できます。例:
- 機密のクライアント名: `"Villa Kaya"`
- 機密のプロジェクトコード: `"internal"`、`"gizli"`

キーワードを追加または削除すると、フィルターは即座に更新されます。

### レイヤー 3 — 手動ファイル/フォルダー非表示

- **単一ファイル:** ファイルを右クリック → **「AI から非表示」** (再度クリックして元に戻すことができます)
- **フォルダー全体:** ソースフォルダーパネルの 3 点メニューから → **「AI から除外」**

### 動作 (技術)

```
ユーザーが AI に質問する
    ↓
RAG パイプラインが検索を開始 (FTS + セマンティック + メタデータ)
    ↓
感度フィルターが起動:
  ✗ rag_excluded = 1 のファイル → スキップ
  ✗ アクティブカテゴリーのキーワードを含むファイル → スキップ
  ✗ カスタムキーワードを含むファイル → スキップ
    ↓
残りのファイルが AI に送信される
```

> **メモ:** フィルターは AI チャットのみに影響します。通常の検索、フィルタリング、詳細パネル、重複ファインダーはフィルターの影響を受けません。

## 10. 重複 & 類似ファイルファインダー

このツールは、アーカイブ内の重複または類似のファイルを検出、比較し、必要に応じて削除します。

> **アクセス — Admin:** すべての機能がアクティブ (スキャン、表示、**削除**)。
> **アクセス — Viewer:** スキャンと表示は可能ですが、**削除操作は admin にのみ開放されています**。

### 10.1 パネルへのアクセス

トップバーの **⎇ (フォーク/枝)** アイコンをクリックします — ゴミ箱アイコンと脳アイコンの間にあります。ホバーすると「重複を検索」と表示されます。

### 10.2 検出モード

4 つの独立したモードを一緒にまたは個別に動作させることができます:

| モード | 何を検出するか? | 動作方式 |
|-----|----------------|----------------|
| **完全一致重複** | 同じ内容のファイル (異なる名前/場所の場合あり) | ファイルハッシュ (SHA 比較) — 即時 |
| **同名** | 異なるフォルダー内の同じファイル名 | ファイル名マッチ — 即時 |
| **ビジュアル類似** | 視覚的に類似する画像 | pHash Hamming distance (64-bit) — ~100ms/1000 画像 |
| **構造類似** | 似たレイヤー/素材/コンテンツ構造 | Jaccard similarity — CAD、3D、ドキュメントのみ |

#### サポートされるファイル種別 (ビジュアル類似)
`JPG · PNG · BMP · WEBP · TIFF · TGA · EXR · HDR · PSD`

#### サポートされるファイル種別 (構造類似)
`DWG · DXF · IFC · MAX · SKP · PDF · DOC/DOCX · XLS/XLSX · PPT/PPTX · RVT`

#### 構造類似性詳細 (ファイル種別ごと、v2.1.2)

| 種別 | 比較されるフィールド |
|-----|------------------------|
| DWG / DXF | レイヤー名 (1.0) + ブロック名 (1.0) + テキスト内容 (0.8) + Xref (0.6) — 重み付き Jaccard |
| IFC | フロア数 + Entity 数 (近接マッチ) + (ある場合) レイヤー名 |
| 3DS MAX | 素材名 (Jaccard) + レンダーエンジンマッチ (+50) + Max バージョンマッチ (+35) |
| SketchUp | コンポーネント名 + レイヤー名 (Jaccard) + SketchUp バージョンマッチ (+35) |
| Revit | フロア名 (Jaccard) + プロジェクト名 (+50) + エリア数の近接性 |
| PDF / DOCX / XLSX... | タイトル (+40) + 著者 (+30) + ページ数 (+30) |

#### 一般基準 (クロスフォーマット事前フィルター)
構造類似と同名モードへのオプションの追加条件:

- **同じファイルサイズ** — 許容差を選択可能: 完全一致、±1 KB、±1%
- **更新日が近い** — N 日のウィンドウ
- **同じフォルダー名** — 親フォルダーの basename マッチ (大文字小文字を区別しない)

これらの条件は `passesGeneralCriteria()` を介してペアごとに適用され、一致しないペアは類似性計算にも入りません (誤検知の削減 + 高速化)。

#### パフォーマンスフィルター
- **最小ファイルサイズ (KB)** — このサイズ未満のファイルは事前フィルターでスキャンプールから除外されます

### 10.3 スコープ選択

パネルが開いたら、スキャンするアーカイブを選択します。利用可能なすべてのアーカイブ (メインアーカイブ、ローカルアーカイブ、追加したカスタムアーカイブ) がタブにリストされます:

| スコープ | 誰がアクセス可能 | 削除権限 |
|--------|-------------------|---------------|
| **メインアーカイブ (shared)** | Admin + Viewer | Admin のみ |
| **ローカルアーカイブ (personal)** | Admin + Viewer | Admin + Viewer |
| **カスタムアーカイブ** | タイプによる (shared = admin のみ書き込み、personal = 所有者が書き込み) | タイプによる |

> **Viewer メモ:** Personal タイプのアーカイブでは削除できます。Shared アーカイブでは admin のみが削除できます。
> アーカイブがまだロードされていない場合、タブは無効に見えます — 先にサイドバーからそのアーカイブに切り替えてください。

### 10.5 類似性しきい値設定

スライダーは **ビジュアル類似** と **構造類似** モードのみに影響します。

```
低しきい値 (50%)         高しきい値 (95%)
◄●─────────────────►     ◄─────────────●►
より多くの結果           少ない、確実な結果
(誤検知のリスク)          (非常に類似のみ)
```

推奨開始値: **75%**。DWG/CAD には 60〜70%、ビジュアルには 85〜90% がより良い結果を生むことがあります。

### 10.6 ハッシュ / メタデータ不足の警告

スキャン結果として 3 つの異なる警告が表示されることがあります:

> ⚠ ハッシュ不足: 23 ファイル — これらは完全一致重複スキャンに含められません。
> ⚠ X ファイルで pHash 不足 — ビジュアル類似スキャンでスキップされました。
> ⚠ X ファイルでメタデータ不足 — 構造類似スキャンでスキップされました。

各警告の隣に:
- **ファイルを表示** — 不足ファイルをリストします
- **これらをスキャン** — これらのファイルのみを再スキャンします (ScanModal pendingRescanPaths 経由)
- **スキップ** — 警告を閉じて、現在の結果で続行します

> **ヒント:** 単一のソースフォルダーを完全に再フレッシュしたい場合は、サイドバー > ソースフォルダー > 3 点 > 「再スキャン」がより実用的です。モーダルが自動的に開き、そのフォルダーのみをスコープで再スキャンします。

### 10.7 リストビュー — 手順

1. **スキャン** ボタンをクリック — 結果はタイプヘッダーごとにグループ化されます
2. グループをクリックして開閉 (▼ / ►)
3. 各グループでは、どのファイルが似ているか、なぜ似ているかが表示されます
4. **最初を残し、他を削除** — 最新のファイルを残し、他をチェックボックスでマークします
5. チェックボックスで希望するファイルを選択
6. **選択したものを削除** — 一括削除には確認が必要

> **メモ:** 削除されたファイルはアーカイブから永続的に削除されます。Ctrl+Z で元に戻すことはできません。

### 10.9 削除フロー

```
チェックボックスを選択  ──►  「選択したものを削除」 ──►  確認ダイアログ
                                                            │
                                            ┌───────────────┴──────────────┐
                                            │  はい                  いいえ │
                                            ▼                               │
                                 ファイルがアーカイブから削除         操作キャンセル
                                 グループが更新される
                                 通知が表示される
```

> **Admin ヒント:** 大規模なアーカイブでは、まず **完全一致重複** モードを単独で実行してください。これらは最も安全な削除候補です。ビジュアル/構造類似の結果は、削除する前に比較ビューで確認してください。

## 11. ファイル関係管理

ファイル間に関係を構築することで、プロジェクトファイルを文脈的に整理できます。

### 関係の種類

| 種類 | 意味 | 自動検出 |
|-----|--------|:---------------:|
| **PDF 出力** | DWG/MAX の PDF バージョン | ✅ (同じ stem、異なる拡張子) |
| **レンダー** | デザインのビジュアライゼーション | ✅ (モデル + ビジュアルマッチ) |
| **バージョン** | 同じファイルの異なるバージョン | ✅ (v1/v2/Rev-A パターン) |
| **プロジェクトグループ** | 同じプロジェクトに属するファイル | 手動 |

### 自動検出
スキャン後、自動関係検出が実行されます:
- `plan.dwg` + `plan.pdf` → PDF 出力 (同じ stem、異なる拡張子)
- `salon.max` + `salon_render.jpg` → レンダー
- **バージョンクラスタリング:** `plan_v1.dwg`、`plan_v2.dwg`、`plan_Rev-A.dwg`、`plan_FINAL.dwg` などのファイルは「バージョン」関係で自動的にリンクされます
  - 認識されるパターン: `_v1`、`_Rev-A`、`_R01`、`_FINAL`、`_DRAFT`、`_eski`、`_yeni`、`(Kopya)`、`(1)/(2)`、末尾の数字
- 自動リンクには `[Otomatik]` ラベルが表示されます

### 手動リンク
1. 詳細パネルの「リンクされたファイル」セクションを開く
2. 「リンク追加」ボタンをクリック
3. 関係の種類を選択
4. リンクするファイルを検索して選択
5. リンクを削除するには行の X ボタンをクリック

### ソースフォルダーメニューからの一括検出
左パネルのソースフォルダーセクションで、任意のフォルダーの 3 点メニューから「リンクをスキャン」を選択すると、そのフォルダー内のすべての関係を再検出できます。

## 12. リクエストシステム (管理者間業務調整)

管理者間の業務調整のためのリクエスト (request) システムが提供されています。

### リクエスト送信
1. メッセージングパネルで新しいメッセージを作成
2. タイプとして **「リクエスト」** を選択
3. 受信者は自動的に非表示になります — リクエストは **すべての管理者** に送信されます
4. 件名と説明を入力して送信

### リクエスト管理
受信したリクエストはメッセージパネルに表示されます:
- **引き受ける** — リクエストを自分の責任にします
- **解除する** — 責任から外れます (リクエストは再びすべての人に開かれます)
- **解決済み** — 引き受けた管理者だけがこのボタンを表示できます

引き受けた管理者の名前はリクエスト行にバッジとして表示されます。

## 13. ゴミ箱と復元

削除されたファイルは永続的に削除されず、まずゴミ箱に移動されます (soft delete)。

### ゴミ箱に移動
- ファイルが削除されると `is_deleted = 1` としてマークされます
- すべてのメタデータ、タグ、関係、プロジェクトステータス情報が保存されます

### 復元
- トップバーのゴミ箱アイコンをクリック
- リストからファイルを選択し、「復元」ボタンをクリック
- ファイルはすべての情報とともにアーカイブに戻ります

### 永久削除
- ゴミ箱で「永久削除」を選ぶと、ファイルはデータベースから完全に削除されます
- 「ゴミ箱を空にする」で、ゴミ箱内のすべてのファイルが永久削除されます
- この操作は元に戻せません

> **メモ:** 再スキャン中に、ゴミ箱内のファイルと同じものがディスク上で見つかった場合、ファイルは自動的にゴミ箱から取り出され、アーカイブに戻されます (ユーザー定義のフィールドは保持されます)。

## 14. アプリケーション終了確認

ウィンドウを **X ボタン** または **Alt+F4** で閉じようとすると、アプリケーションは確認ダイアログを表示します。

- **「終了」** — アプリケーションを閉じます (進行中の操作は終了されます)
- **「キャンセル」** — アプリケーション内に留まります

この保護により、長時間のスキャンやダウンロード操作が誤って中断されるのを防ぎます。
